# AI 助理本地規則（AGENTS.md — 跨 AI 通用標準）

本檔為本 repo 的最高優先 AI 行為規範。適用於所有 AI 編碼助理
（Claude、GPT/Codex、Gemini、Copilot、Cursor 等）。與其他指示衝突時，以本檔為準。
一律使用**繁體中文**回覆。

---

## 一、誠實回報鐵律（最高優先）

**未經驗證，禁止宣告完成。** 「好了」「已完成」「已修復」這類話，只有在你
「實際執行過驗證步驟、且親眼看到成功輸出」之後才准說。

1. 回報任務狀態時，**只能**使用以下三種標記，不得使用模糊說法：
   - `✅ 已完成（已驗證）`：附上驗證指令與**實際輸出**（非想像中的輸出）。
   - `⚠️ 已修改（未驗證）`：說明改了什麼、為何無法驗證（如環境缺工具）。
   - `❌ 失敗／受阻`：照實貼出錯誤訊息，說明卡在哪、需要什麼。
2. 每個 `✅` 都必須附**證據**，至少一項：
   - 執行過的指令 + 關鍵輸出（原文引用，不得改寫美化）
   - `git diff --stat` 或變更檔案清單（`檔名:行號`）
   - 針對變更點的 grep/讀檔結果，證明變更確實存在於檔案中
3. 測試或指令失敗時：**原樣回報**，禁止淡化（不得說「大致成功」「應該沒問題」）。
4. 不確定的事就說「不確定」，並說明會如何查證。**禁止猜測性肯定**、
   禁止捏造不存在的檔案、函式、URL、API、輸出結果。
5. 只完成一部分就說一部分：「10 項中完成 7 項，剩餘 3 項是…」，
   禁止以偏概全地宣稱整體完成。
6. 被要求做 N 件事，結束前逐項對照原始需求清點，漏做的明確列出。

## 二、完成的定義（Definition of Done）

一項修改要標 `✅`，必須依序做到：

1. **改前**：先讀過要改的區段（局部讀取即可），確認理解現狀。
2. **改後**：`git diff` 確認 diff 內容與意圖一致、無誤傷其他區塊。
3. **能執行就執行**：有可跑的驗證方式就必須跑（見下方本 repo 驗證表）。
   環境跑不了 → 誠實標 `⚠️ 已修改（未驗證）`，不准硬標 `✅`。
4. **驗證必須親自執行、且執行兩輪**：
   - **第一輪（正向）**：照速查表實際跑驗證指令，親眼看到成功輸出。
   - **第二輪（挑錯）**：換角度主動找不合理——假設「裡面一定有錯」重讀整個
     diff，逐項檢查：與原始需求不符？邏輯矛盾？編號／連結／引用失效？誤傷
     無關區塊？邊界情況沒顧到？找到問題就修掉並重跑第一輪。
   - 兩輪都要寫進回報：第二輪即使沒找到問題，也要列出「檢查了哪些角度」。
   - 禁止拿記憶中、上一輪任務的舊輸出充當本次證據；每個 ✅ 的證據都必須
     來自本次任務中親自執行的結果。
5. **引用實際輸出**作為證據寫進回報。

### 本 repo 驗證方式速查

| 目標 | 驗證方式 |
|---|---|
| `index.html` 等 HTML | `python3 -c "from html.parser import HTMLParser; HTMLParser().feed(open('index.html',encoding='utf-8').read()); print('PARSE OK')"`；再 grep 確認變更點存在 |
| `fetch_news.ps1` | 有 `pwsh` 才能語法檢查：`pwsh -NoProfile -c "[void][ScriptBlock]::Create((Get-Content -Raw fetch_news.ps1)); 'SYNTAX OK'"`；無 pwsh 一律標 `⚠️ 未驗證` |
| `.github/workflows/*.yml` | `python3 -c "import yaml,sys; yaml.safe_load(open(sys.argv[1])); print('YAML OK')" <檔案>` |
| JSON（`vercel.json` 等） | `python3 -m json.tool <檔案> > /dev/null && echo JSON OK` |
| 純文字/文件 | `git diff` + 讀取變更段落確認 |

## 三、省 Token 規則（硬性）

1. **先講結論**：回覆第一句就是結果或答案，細節放後面。
2. **不貼大段程式碼**：引用 `檔名:行號` 即可；只有新寫的關鍵片段才貼。
3. **不逐步實況轉播**：不要每個小動作都寫一段報告；工作中只在轉折點簡短說明。
4. **大檔案禁止整檔讀取**（本 repo 地雷）：
   - `.translate_cache.json`（約 6.7MB）：**永遠禁止讀取**，它只是翻譯快取。
   - `index.html`（約 390KB）、`carbon-plate-cae-simulator.html`（約 670KB）、
     `teisale-prototype.html`（約 240KB）、`fetch_news.ps1`（約 176KB）：
     一律先 grep 定位，再以 offset/limit 局部讀取需要的區段。
5. **不重複勞動**：已讀過、已確認過的內容不重讀；已回答過的不重述。
6. **不做沒被要求的事**：不順手重構、不加沒人要的功能、不改無關檔案。
   發現問題可以「一句話提及」，但不動手，除非使用者同意。
7. 簡單問題用短答，不要為了看起來認真而灌水（不濫用標題、表格、條列）。

## 四、工作流程

1. **開工前先報告工作流程（強制）**：動手改任何東西之前，必須先用 3～5 行報告：
   - **目標**：這次要達成什麼（對照使用者原始需求）。
   - **要動的檔案**：列出檔名；不在清單上的檔案不得修改。
   - **步驟**：依序要做哪幾步。
   - **驗證方式**：完成後打算用什麼指令證明（對照第二章速查表）。
   報告完才能開始執行；執行中若發現需偏離計畫（多改檔案、換方法），先說明再繼續。
   純問答、沒有要改檔案的任務不需要此報告。
2. 一次任務一個目標；改動保持最小必要範圍。
3. commit 訊息用中文或英文皆可，但要具體描述「改了什麼、為什麼」。
4. 推送前必跑：`git status` + `git diff --stat`，確認沒有夾帶垃圾檔案。
5. 破壞性操作（刪檔、覆蓋、force push、對外發布）前先確認或詢問。
6. 禁止 force push `main`；不建 PR 除非使用者明確要求。

## 五、違規自查

回覆送出前自問：
- 我說的每個「完成」都有實際輸出當證據嗎？證據是本次親自跑出來的嗎？
- 我有沒有親自跑完兩輪驗證（第一輪正向、第二輪挑錯）並回報第二輪查了什麼？
- 我有沒有把「我改了程式碼」偷換成「功能正常運作」？（兩者不同！）
- 這則回覆有沒有可以刪掉一半而不損失資訊的部分？

詳細驗證流程見 `.claude/skills/verify-done/SKILL.md`（跨 AI 皆可閱讀遵循）。
